# 🏦 Fintech Ledger Engine

> Высоконагруженный банковский движок, спроектированный с упором на **отказоустойчивость**, **консистентность данных** и **безопасность**. Реализует микросервисную архитектуру для обработки финансовых транзакций, защищённую от Race Condition и дублирующихся запросов.

---

## 🛠 Технологический стек

| Категория | Технологии |
|---|---|
| **Core** | Java 21, Spring Boot 3.x, Maven |
| **Database** | PostgreSQL + Liquibase (версионирование схемы) |
| **Reliability & Caching** | Redis (хранение ключей идемпотентности) |
| **Security** | Spring Security + JWT (Stateless), BCrypt |
| **Testing** | Testcontainers (интеграционные тесты с реальным Postgres в Docker) |
| **Mapping** | MapStruct (высокопроизводительный маппинг Entity → DTO) |

---

## 🏗 Ключевые архитектурные решения

### 1. Механизм Идемпотентности (Idempotency)

Для защиты от повторных списаний (**Double Spend**) при сбоях сети реализована поддержка заголовка `Idempotency-Key`:

- При поступлении запроса сервис проверяет наличие ключа в **Redis**
- Если ключ найден — возвращается результат предыдущей операции **без повторной обработки**
- Это гарантирует безопасность транзакций даже при агрессивных ретраях со стороны клиента

### 2. Консистентность и борьба с Race Condition

- **Optimistic Locking** — использование поля `@Version` в сущности `Account` предотвращает проблему **"Lost Update"**. Если две транзакции пытаются изменить баланс одновременно, одна из них завершится ошибкой, не нарушив целостность данных
- **Transactional Integrity** — все переводы атомарны. При возникновении любой ошибки (несоответствие валют, нехватка средств) происходит полный **Rollback**

### 3. Безопасность и Ownership Check

- **Custom Security Context** — в качестве `Principal` используется кастомная сущность `User`, что позволяет инжектить текущего пользователя напрямую через `@AuthenticationPrincipal`
- **Проверка владения** — реализована жёсткая проверка прав на уровне сервисов: пользователь может управлять только своими счетами, попытки доступа к чужим данным блокируются (`403 Forbidden`)

## 📊 Модель данных

| Сущность | Описание |
|---|---|
| **User** | Система ролей, поддержка системных аккаунтов для внутренних нужд |
| **Account** | Мультивалютность, хранение баланса с защитой от конкурентного доступа |
| **Transaction** | Детальный аудит состояний (`PENDING`, `COMPLETED`, `FAILED`) и логирование всех участников перевода |

## 🚀 API Эндпоинты

| Метод | Эндпоинт | Описание | Обязательные заголовки |
|---|---|---|---|
| `POST` | `/api/auth/login` | Аутентификация и выдача JWT | — |
| `POST` | `/api/transfers` | Перевод средств между счетами | `Authorization`, `Idempotency-Key` |
| `GET` | `/api/accounts/{id}/transactions` | История транзакций (пагинация/фильтрация) | `Authorization` |

> **Security Note:** Внутренние ID базы данных скрыты. В API используются только публичные номера счетов, маппинг которых настроен через **MapStruct**.

---

## 🧪 Качество и тестирование

В проекте используется подход **Testcontainers** — запуск тестов в окружении, максимально приближённом к продакшену (реальный **PostgreSQL** и **Redis** в Docker). Это исключает ошибки, специфичные для H2 или других in-memory решений.
